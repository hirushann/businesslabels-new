# Scout Index Frontend Guide

This guide is for frontend work that needs to read directly from Elasticsearch, especially from Next.js server code. Laravel remains the source of truth for index names, document shape, and indexing behavior.

Do not call Elasticsearch from browser components. Keep ES hostnames and credentials in Next.js server-only environment variables, route handlers, server actions, or backend-for-frontend code.

## Scout Setup

- Scout driver: `elastic`
- Index prefix: `SCOUT_PREFIX`, currently `business_labels_` in `.env.example`
- Elasticsearch connection: `ELASTIC_HOST`, `ELASTIC_PORT`, `ELASTIC_SCHEME`, `ELASTIC_USERNAME`, `ELASTIC_PASSWORD`
- Laravel index gateway: `App\Services\ElasticCatalogSearchGateway`
- Reindex command: `php artisan app:reindex-elasticsearch`

Scout index names are generated by each model's `searchableAs()` method. Always include the configured prefix when calling Elasticsearch directly.

## Indexes

| Purpose | Model | Index | Scout document ID |
| --- | --- | --- | --- |
| Simple products | `App\Models\Product` | `business_labels_catalog_products_simple` | `product_{id}` |
| Group products | `App\Models\GroupProduct` | `business_labels_catalog_products_simple` | `group_product_{id}` |
| Variable products | `App\Models\MasterProduct` | `business_labels_catalog_products_variable` | `{id}` |
| Printers | `App\Models\Post` where `post_type = printer` | `business_labels_catalog_printers` | `printer_{id}` |
| Materials | `App\Models\Material` | `business_labels_catalog_materials` | `material_{id}` |

Group products intentionally share the simple products index. Use `product_type` or `is_group_product` to distinguish them.

Important differences:

- `Product` stores many localized fields as arrays, for example `title: string[]`.
- `MasterProduct` stores `name` and `title` as strings, but stores other localized fields like `slug` and `description` as arrays.
- `GroupProduct` stores `name` and `title` as strings, uses `product_type: "group"`, and has `is_group_product: true`.
- `GroupProduct` prices are derived from components: `price` is the final discounted price, `original_price` is the subtotal before discount, and `discount` is the percentage.
- For product detail pages, prefer `api_path_by_id`, `api_path_by_slug`, or `frontend_path` when available instead of reconstructing paths in the frontend.
- Product facets use canonical fields only: `catalog_brand`, `catalog_material_code`, `catalog_material`, and `compatible_brands`. Use `catalog_brand.keyword` for exact brand filtering and aggregations.
- Vanilo properties are the source of truth for catalog attributes. Promoted properties are intentionally not duplicated in `properties`.
- `material_ids` mirrors `material_id` as an array so frontend Elasticsearch filters can use the same `terms` pattern as `printer_ids`.

## Printer Documents

Printers are indexed from `App\Models\Post` records where `post_type = printer` and `status = published`.

```ts
type PrinterSearchDocument = {
  id: number;
  title: string[];
  title_sort: string;
  subtitle?: string[];
  slug: string[];
  excerpt?: string[];
  content?: string[];
  status: 'published';
  main_image?: string | null;
  properties?: Record<string, string[]>;
  product_ids?: number[];
  created_at_timestamp: number;
};
```

Printer `properties` are dynamic values from `post_meta`. The index excludes `subtitle`, `featured`, and `printer_url` from `properties`.

Common printer filters:

- `properties.kern.keyword`
- `properties.druktype.keyword`
- `properties.detectie.keyword`
- `properties.buiten_diameter.keyword`
- `properties.width.keyword`
- `product_ids`


Important relationship fields:

- `product_ids` contains simple product IDs from `products.material_id`.
- `active_product_ids` and `products` match the material detail API behavior: only active simple products, ordered by name.
- `master_product_ids` contains variable product IDs from `master_products.material_id`.
- Product documents expose `material_ids`, so the frontend can filter product indexes with `{ terms: { material_ids: [materialId] } }`.
- Keep simple and variable product IDs separate because both tables can contain the same numeric ID.
- `title`, `subtitle`, `slug`, `description`, labels, `categories`, `specifications`, `spec_sheet_url`, `has_uploaded_spec_sheet`, `products`, and `products_count` are included so frontend code can migrate from `GET /api/materials/{id}` to Elasticsearch without losing current fields.
- Use `*_locales` fields or `translations` for cross-locale search/display. The scalar fields are resolved using Laravel's main locale at index time.

## Query Patterns

Use `_source` to keep responses small. Hydrate full product or printer details from Laravel APIs when the page needs relational data, pricing rules, auth-specific values, or fresh business logic.

### Product Listing Search

This mirrors the Laravel catalog search behavior in `ProductCatalogService`.

```ts
const payload = {
  index: [
    'business_labels_catalog_products_simple',
    'business_labels_catalog_products_variable',
  ],
  body: {
    track_total_hits: true,
    from: (page - 1) * perPage,
    size: perPage,
    _source: [
      'id',
      'model_id',
      'product_type',
      'is_group_product',
      'title',
      'slug',
      'sku',
      'price',
      'stock',
      'in_stock',
      'main_image',
      'frontend_path',
    ],
    query: {
      bool: {
        must: search
          ? [{
              multi_match: {
                query: search,
                fields: [
                  'article_number^7',
                  'title^5',
                  'name^4',
                  'slug^4',
                  'sku^6',
                  'variant_skus^6',
                  'catalog_brand^2',
                  'catalog_material_code^2',
                  'catalog_material^2',
                  'compatible_brands',
                  'properties.printmethode',
                  'properties.afwerking',
                  'properties.lijm',
                  'properties.detectie',
                  'excerpt^2',
                  'description',
                  'content',
                  'product_information',
                ],
                type: 'bool_prefix',
                operator: 'and',
              },
            }]
          : [{ match_all: {} }],
        filter: [
          { term: { 'state.keyword': 'active' } },
          { range: { stock: { gt: 0 } } },
        ],
      },
    },
    sort: [
      { created_at_timestamp: { order: 'desc', unmapped_type: 'long' } },
    ],
  },
};
```

#### Category Hierarchy Filtering

Filter by hierarchical category structure using the `categories` nested field:

```ts
// Filter by specific category (any level)
const filters = [
  {
    nested: {
      path: 'categories',
      query: {
        bool: {
          must: [
            { term: { 'categories.id': 123 } },
          ],
        },
      },
    },
  },
];

// Filter by category slug across all levels
const filters = [
  {
    nested: {
      path: 'categories',
      query: {
        terms: { 'categories.slug.keyword': ['labels', 'thermal-labels'] },
      },
    },
  },
];

// Filter by category level (root = 0, children = 1, grandchildren = 2)
const filters = [
  {
    nested: {
      path: 'categories',
      query: {
        term: { 'categories.level': 1 },
      },
    },
  },
];

// Filter by breadcrumb (includes all ancestors)
const filters = [
  {
    nested: {
      path: 'categories',
      query: {
        terms: { 'categories.breadcrumb_ids': [1, 2, 3, 12, 34] },
      },
    },
  },
];
```

Supported catalog meta filters:

| Frontend query | ES exact field | ES range field | Type |
| --- | --- | --- | --- |
| `afwerking` | `properties.afwerking.keyword` |  | multi-select |
| `lijm` | `properties.lijm.keyword` |  | multi-select |
| `brand` | `catalog_brand.keyword` |  | multi-select |
| `materiaal-code` | `catalog_material_code` |  | multi-select |
| `materiaal` / `material` | `catalog_material` |  | multi-select |
| `printmethode` | `properties.printmethode.keyword` |  | multi-select |
| `breedte` | `properties.breedte.keyword` | `property_numbers.breedte` | range |
| `hoogte` | `properties.hoogte.keyword` | `property_numbers.hoogte` | range |
| `kern` | `properties.kern.keyword` | `property_numbers.kern` | range |
| `buiten-diameter` | `properties.buiten-diameter.keyword` | `property_numbers.buiten-diameter` | range |
| `detectie` | `properties.detectie.keyword` |  | multi-select |
| `merken` | `compatible_brands` |  | multi-select |

The Laravel API also accepts aliases like `width`, `height`, `core`, `outer_diameter`, `druktype`, `print_method`, `glue`, and `adhesive`. When calling Elasticsearch directly, prefer the canonical ES field names above.

Removed legacy product-index fields:

- `make` is not indexed for filtering. Product brand comes from the Vanilo `brand` property when present, falls back to the product brand value on the product record, and is indexed as `catalog_brand` with aggregations on `catalog_brand.keyword`.
- `material_title` and `material_slug` are not indexed. Material type/name is indexed as `catalog_material`; material category filters use `material_taxon_ids` / `material_taxon_slugs`.
- `properties.brand`, `properties.materiaal-code`, `properties.materiaal`, and `properties.merken` are promoted into canonical fields and intentionally omitted from `properties`.
- The compact `compatibility` object is not indexed. Use `properties`, `property_numbers`, `compatible_brands`, and `printer_ids`.

## Next.js Server Client

Keep this client in server-only code, for example `src/lib/server/elasticsearch.ts`.

```ts
import 'server-only';

const endpoint = process.env.ELASTICSEARCH_URL;
const username = process.env.ELASTICSEARCH_USERNAME;
const password = process.env.ELASTICSEARCH_PASSWORD;

export async function searchElasticsearch<TDocument>(payload: {
  index: string | string[];
  body: Record<string, unknown>;
}) {
  if (!endpoint) {
    throw new Error('Missing ELASTICSEARCH_URL');
  }

  const authHeader =
    username && password
      ? { Authorization: `Basic ${Buffer.from(`${username}:${password}`).toString('base64')}` }
      : {};

  const index = Array.isArray(payload.index)
    ? payload.index.join(',')
    : payload.index;

  const response = await fetch(`${endpoint}/${index}/_search`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      ...authHeader,
    },
    body: JSON.stringify(payload.body),
    cache: 'no-store',
  });

  if (!response.ok) {
    throw new Error(`Elasticsearch search failed: ${response.status}`);
  }

  return response.json() as Promise<{
    hits: {
      total: { value: number; relation: string } | number;
      hits: Array<{
        _id: string;
        _index: string;
        _score?: number;
        _source: TDocument;
      }>;
    };
    aggregations?: Record<string, unknown>;
  }>;
}
```

## Response Normalization

Elasticsearch returns `_id`, `_index`, `_score`, and `_source`. UI components should usually consume normalized objects:

```ts
function normalizeHits<TDocument>(response: {
  hits: { hits: Array<{ _id: string; _index: string; _source: TDocument }> };
}) {
  return response.hits.hits.map((hit) => ({
    scoutId: hit._id,
    index: hit._index,
    ...hit._source,
  }));
}
```

For simple and group products, prefer `_source.model_id` when present. `_id` is prefixed by Scout and is not always a plain integer.

## When To Use Laravel Instead

Call Laravel APIs instead of Elasticsearch when the feature needs:

- Authentication, customer-specific pricing, favorites, carts, or order context
- Writes or mutations
- Policy checks or hidden business rules
- Fully hydrated relationships
- Guaranteed fresh data immediately after a write

Use Elasticsearch directly when the feature needs fast catalog search, filtering, sorting, pagination, or lightweight autocomplete from public index data.
